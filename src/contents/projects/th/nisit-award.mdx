## 🎯 วัตถุประสงค์

โปรเจกต์นี้เป็นส่วนหนึ่งของรายวิชา
**01418441-65 เว็บเทคโนโลยีและเว็บบริการ** และ **01418472-65 การบูรณาการกระบวนการเชิงอไจล์และเดฟอ็อปส์** ภาควิชาวิทยาการคอมพิวเตอร์ มหาวิทยาลัยเกษตรศาสตร์ โดยได้รับโจทย์จากทางมหาวิทยาลัย (กองพัฒนานิสิต) ที่ต้องการนำเทคโนโลยีเข้ามาช่วยสร้างแพลตฟอร์มเพื่ออำนวยความสะดวกในการจัดการและตรวจสอบใบสมัครของนิสิตผู้มีผลงานดีเด่นในด้านต่าง ๆ เพื่อสร้างช่องทางที่ชัดเจน เป็นระบบ และโปร่งใสในการคัดเลือก จึงเป็นที่มาของการพัฒนา **Nisit Award** หรือ ระบบจัดการใบสมัครนิสิตผู้มีผลงานดีเด่น

## 💻 ฟีเจอร์หลักของระบบ

ระบบนี้ได้แบ่งสิทธิ์การเข้าถึงออกเป็น 6 บทบาทหลัก เพื่อความปลอดภัยและครอบคลุมทุกกระบวนการทำงาน ดังนี้:

### 1. นิสิต / นักศึกษา

- จัดการและติดตามสถานะใบสมัครรางวัลนิสิตดีเด่นของตนเอง

### 2. หัวหน้าภาควิชา

- ตรวจสอบ ติดตามสถานะ และพิจารณา (อนุมัติหรือปฏิเสธ) ใบสมัครของนิสิตภายในภาควิชาและวิทยาเขตที่สังกัด

### 3. รองคณบดี

- ตรวจสอบ ติดตามสถานะ และพิจารณา (อนุมัติหรือปฏิเสธ) ใบสมัครของนิสิตภายในคณะและวิทยาเขตที่สังกัด

### 4. คณบดี

- ตรวจสอบ ติดตามสถานะ และพิจารณา (อนุมัติหรือปฏิเสธ) ใบสมัครของนิสิตภายในคณะและวิทยาเขตที่สังกัด

### 5. กองพัฒนานิสิต / ผู้ดูแลระบบ

- บริหารจัดการบัญชีผู้ใช้งาน คณะ และภาควิชาทั้งหมดภายในวิทยาเขตที่สังกัด
- จัดการรอบการรับสมัคร (1 รอบ = 1 ปีการศึกษาและภาคเรียน) และหมวดหมู่ของรางวัลในแต่ละรอบ
- ตรวจสอบ ติดตามสถานะ และพิจารณาใบสมัครของนิสิตภายในวิทยาเขต
- อัปโหลดเอกสารที่ได้รับการรับรองจากอธิการบดี เพื่อดำเนินการปิดรอบการรับสมัครปัจจุบันและประกาศรายชื่อผู้ได้รับรางวัล

### 6. คณะกรรมการ

- ตรวจสอบ ติดตามสถานะ และพิจารณา (อนุมัติหรือปฏิเสธ) ใบสมัครของนิสิตภายในวิทยาเขตที่สังกัด

> 💡 **หมายเหตุ:** ผู้ใช้งานทุกบทบาท (ยกเว้นกองพัฒนานิสิต) จะใช้งานผ่านหน้าเว็บที่พัฒนาด้วย Svelte ในขณะที่บทบาท "กองพัฒนานิสิต" จะดำเนินการผ่านระบบหลังบ้านที่พัฒนาด้วย Laravel

## 🔁 กระบวนการทำงานของทีม

ทีมของเราประยุกต์ใช้วิธีการทำงานแบบ **Agile Methodology (Scrum)** เพื่อให้การพัฒนาระบบมีความยืดหยุ่นและตอบรับต่อการเปลี่ยนแปลงได้ดี โดยใช้ **Jira** เป็นเครื่องมือหลักในการติดตามงาน (Backlog Items) ของสมาชิกแต่ละคน

- **Sprint Planning:** การทำงานแบ่งออกเป็น 5 Sprint โดยแต่ละ Sprint จะมีระยะเวลาประมาณ 2-3 สัปดาห์
- **Sprint Retrospective & Refinement:** หลังจบแต่ละ Sprint ทีมจะจัดประชุม Retrospective ทุกวันจันทร์ เวลา 15:00 น. ในรูปแบบ Onsite เพื่อจัดทำ Burndown Chart และร่วมกันวิเคราะห์ถึงสิ่งที่ทำได้ดีรวมถึงสิ่งที่ต้องปรับปรุง นอกจากนี้ยังมีการทำ Backlog Refinement เพื่อลงรายละเอียดงานที่ยังไม่ชัดเจน หรือปรับแผนรับมือกับการเปลี่ยนแปลง Requirement (ทั้ง Major และ Minor Change) ก่อนที่จะเริ่มวางแผนใน Sprint ถัดไป

## 💡 สิ่งที่ได้เรียนรู้

- **การ Merge โค้ดอย่างสม่ำเสมอช่วยลดปัญหา Merge Conflict:** การหมั่นนำโค้ดมารวมกัน ให้บ่อยที่สุดเท่าที่จะเป็นไปได้ (เช่น อย่างน้อยวันละ 1 ครั้ง) ช่วยลดความซับซ้อนและป้องกันความผิดพลาดร้ายแรงจากการรวมโค้ดที่มีความขัดแย้งกันได้อย่างมีประสิทธิภาพ
- **ความสำคัญของการทดสอบระบบ (Testing):** ระหว่างการพัฒนา ทีมพบข้อผิดพลาดที่เกิดจากการทดสอบที่ไม่ครอบคลุมเพียงพอ ซึ่งส่งผลกระทบให้สมาชิกคนอื่นไม่สามารถทำงานต่อได้ และอาจบั่นทอนความเชื่อมั่นของผู้มีส่วนได้ส่วนเสีย ระหว่างการนำเสนองาน เราจึงตระหนักว่าการทดสอบควรเป็นกระบวนการที่ต้องให้ความสำคัญเป็นอันดับต้น ๆ และต้องเผื่อเวลาไว้เสมอ
- **การเตรียมความพร้อมสำหรับการ Deployment:** ในขั้นตอนการทำ UAT (User Acceptance Testing) หรือการนำขึ้น Production มักจะมีปัญหาที่ไม่เคยพบใน Development Environment ปรากฏขึ้นเสมอ เนื่องจากการตั้งค่าสภาพแวดล้อมที่แตกต่างกัน การเผื่อเวลาสำหรับการหาสาเหตุ และแก้ไขปัญหาในสภาพแวดล้อมจริงจึงเป็นสิ่งที่ขาดไม่ได้

## 👾 วิธีติดตั้งและรันโปรแกรม

### Svelte

1. Clone โปรเจกต์ส่วน Svelte จาก GitHub ลงมาที่เครื่อง

```bash
git clone https://github.com/thg1rb/p2-final-project-hbd-feel.git
```

2. เข้าสู่โฟลเดอร์ของโปรเจกต์

```bash
cd p2-final-project-hbd-feel
```

3. คัดลอกไฟล์ตั้งค่าเริ่มต้น (Environment Variables) เพื่อเตรียมการตั้งค่าระบบ

```bash
cp .env.example .env
```

4. รัน Container ผ่าน Docker แบบพื้นหลังเพื่อเริ่มต้นระบบ

```bash
docker compose up -d
```

### Laravel

1. Clone โปรเจกต์ส่วน Laravel จาก GitHub ลงมาที่เครื่อง

```bash
git clone https://github.com/thg1rb/p2-final-project-backend-hbd-feel.git
```

2. เข้าสู่โฟลเดอร์ของโปรเจกต์

```bash
cd p2-final-project-backend-hbd-feel
```

3. คัดลอกไฟล์ตั้งค่าเริ่มต้นสำหรับฝั่ง Laravel

```bash
cp .env.example .env
```

4. ใช้ Docker รัน Composer เพื่อติดตั้ง Dependencies ของ Laravel โดยไม่ต้องติดตั้ง PHP หรือ Composer ลงในเครื่องโดยตรง

```bash
docker run --rm \
    -u "$(id -u):$(id -g)" \
    -v "$(pwd):/app" \
    -w /app \
    composer:latest \
    install --ignore-platform-reqs
```

5. เริ่มต้นการทำงานของ Container ทั้งหมดที่กำหนดไว้ใน Laravel Sail (เช่น Server, Database, Redis)

```bash
sail up -d
```

6. สร้าง Application Key ใหม่สำหรับใช้ในการเข้ารหัสข้อมูลเพื่อความปลอดภัยของระบบ

```bash
sail artisan key:generate
```

7. รันคำสั่งสร้างตารางในฐานข้อมูลใหม่ทั้งหมด พร้อมกับสร้างข้อมูลจำลองเบื้องต้น (Seeding)

```bash
sail artisan migrate:fresh --seed
```

8. ติดตั้งแพ็กเกจที่จำเป็นสำหรับการจัดการ Assets ฝั่ง Frontend ภายใน Laravel (ผ่านเครื่องมือ Bun)

```bash
sail bun install
```

9. รันคำสั่งสำหรับคอมไพล์ Frontend assets (เช่น TailwindCSS) ในโหมด Development

```bash
sail bun dev
```

10. เปิด Terminal หน้าต่างใหม่ เพื่อรันระบบคิวสำหรับประมวลผลการทำงานเบื้องหลัง (Background Job) เช่น การส่งอีเมล

```bash
sail artisan queue:work
```
